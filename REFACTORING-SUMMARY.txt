✅ PESQUISA REALIZADA:
   • TinyDB (puro Python, muito leve)
   • Mongita ⭐ RECOMENDADO (MongoDB-compatible, escalável)
   • UnQLite (obsoleto desde 2014)
   • LMDB (rápido mas chave-valor apenas)
   • RocksDB (muito pesado para Docker leve)
   • PoloDB (novo, menos suporte)

✅ SOLUÇÃO ESCOLHIDA: MONGITA
   ├─ MongoDB-compatible (PyMongo API)
   ├─ Embedded como SQLite (sem servidor)
   ├─ BSON binary format (20% mais compacto que JSON)
   ├─ Docker: ~180-200MB (vs 300MB+ com SQLite+ORM)
   ├─ Schema-less (flexível para novos campos)
   ├─ Escalável para MongoDB cloud
   └─ Dependências mínimas: pymongo + sortedcontainers

⚡ VANTAGENS PRINCIPAIS
═══════════════════════════════════════════════════════════════════════════

1. FLEXIBILIDADE DE SCHEMA
   Pesquisador solicita "origem_do_artigo"?
   Antes: ALTER TABLE + migration + alembic upgrade
   Depois: db['referencias'].insert_one({'origem_do_artigo': '...'})
   → Zero downtime, código simples!

2. PERFORMANCE DE LEITURA
   Uma query retorna TUDO: referência + espécies + comunidades + estudo
   SQLite: SELECT a.* JOIN DadosEstudo JOIN ArtigoEspecie JOIN... (complexo)
   Mongita: db['referencias'].find_one({'_id': id}) (simples)

3. DOCKER LEVE
   SQLite + SQLAlchemy: 300-350 MB
   Mongita + PyMongo: 180-200 MB
   → Economia: 40-50%! Importante para UNRAID.

4. ESCALABILIDADE FUTURA
   Mongita (hoje): Embedded, single-machine
   MongoDB (amanhã): Cloud, multi-machine
   → Código: 100% compatível! Apenas mude Client.

5. ESTRUTURA NATURAL PARA ETNOBOTÂNICA
   Pesquisa é hierárquica: Referência → Espécies → Comunidades → Usos
   JSON reflete exatamente essa hierarquia
   → Agregação natural reduz complexity!
